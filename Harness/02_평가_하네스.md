# 02. 평가 하네스 (Evaluation Harness)

> LLM의 성능을 표준화된 방식으로 측정하고 비교하는 평가 프레임워크를 다룹니다.

---

## 왜 평가 하네스가 필요한가

LLM은 같은 프롬프트에도 매번 다른 답변을 낼 수 있고, 벤치마크 결과를 직접 비교하려면 **동일한 조건(프롬프트 포맷, 샘플링 방식, 메트릭 계산 방법)**이 보장되어야 합니다.

평가 하네스는 이 조건을 자동화하고 재현 가능하게 만드는 **표준 실행 환경**입니다.

---

## 주요 평가 하네스 도구

### EleutherAI lm-evaluation-harness

가장 널리 사용되는 오픈소스 LLM 평가 프레임워크입니다.

```bash
pip install lm-eval

# Hugging Face 모델 평가
lm_eval --model hf \
  --model_args pretrained=meta-llama/Llama-2-7b-hf \
  --tasks hellaswag,arc_easy \
  --device cuda:0 \
  --batch_size 8
```

**지원 태스크 예시**:

| 태스크 | 측정 대상 |
|--------|-----------|
| `hellaswag` | 상식 추론 |
| `arc_easy` / `arc_challenge` | 과학 지식 |
| `mmlu` | 다분야 지식 (57개 과목) |
| `gsm8k` | 수학적 추론 |
| `truthfulqa` | 사실 정확성 |
| `humaneval` | 코드 생성 |

### OpenAI Evals

OpenAI가 모델 품질을 측정하기 위해 공개한 평가 프레임워크.

```yaml
# eval 정의 파일 예시 (YAML)
id: my-custom-eval
description: "사용자 정의 평가"
metrics: [accuracy]

samples:
  - input: "대한민국의 수도는?"
    ideal: "서울"
```

```bash
pip install evals
oaieval gpt-4o my-custom-eval
```

---

### HELM (Holistic Evaluation of Language Models)

Stanford CRFM에서 개발한 종합 평가 프레임워크. **공정성, 효율, 정확성**을 다각도로 측정합니다.

- 42개 시나리오, 59개 메트릭
- 정확도뿐 아니라 **편향(bias), 독성(toxicity), 보정(calibration)** 측정
- [crfm.stanford.edu/helm](https://crfm.stanford.edu/helm) 에서 결과 공개

## 평가 하네스의 핵심 구성

```
┌─────────────────────────────────────────────────────┐
│                   평가 하네스 구조                   │
│                                                     │
│  데이터셋     →   프롬프트 빌더  →  모델 호출         │
│  (벤치마크)        (few-shot 등)     (API / 로컬)    │
│                                         ↓           │
│  리포트       ←   메트릭 계산   ←  응답 파싱          │
│  (HTML/JSON)      (정확도 등)       (정규화)         │
└─────────────────────────────────────────────────────┘
```

### 프롬프트 포맷 방식

| 방식 | 설명 | 사용 시점 |
|------|------|-----------|
| `0-shot` | 예시 없이 질문만 | 기본 능력 측정 |
| `few-shot` | N개 예시 포함 | 문맥 학습(ICL) 능력 측정 |
| `chain-of-thought` | "단계적으로 생각하라" 지시 추가 | 추론 능력 측정 |

## 커스텀 평가 만들기

`lm-evaluation-harness` 기준 커스텀 태스크 예시:

```python
# tasks/my_task.py
from lm_eval.api.task import Task
from lm_eval.api.instance import Instance

class MyTask(Task):
    VERSION = 0
    DATASET_PATH = "my_dataset"

    def has_training_docs(self): return False
    def has_validation_docs(self): return True

    def validation_docs(self):
        return self.dataset["validation"]

    def doc_to_text(self, doc):
        return f"질문: {doc['question']}\n답변:"

    def doc_to_target(self, doc):
        return f" {doc['answer']}"

    def process_results(self, doc, results):
        pred = results[0].strip()
        gold = doc["answer"]
        return {"accuracy": int(pred == gold)}

    def aggregation(self):
        return {"accuracy": "mean"}
```

## 평가 결과 해석 주의사항

```
⚠️ 흔한 함정

1. 데이터 오염(Data Contamination)
   - 훈련 데이터에 벤치마크 문제가 포함된 경우
   - 높은 점수가 진짜 실력인지 확인 필요

2. 프롬프트 민감성
   - 동일 태스크라도 프롬프트 포맷에 따라 점수 차이가 큼
   - 비교 시 반드시 동일 포맷 사용

3. 메트릭 편향
   - Accuracy만으로는 불충분 — calibration, robustness 함께 확인
   
4. 단일 벤치마크 의존
   - 여러 벤치마크를 조합해서 종합적으로 판단
```

## 실무 적용 예시

### 모델 업그레이드 전 회귀 테스트

```bash
# 기존 모델 평가
lm_eval --model openai-chat-completions \
  --model_args model=gpt-4o \
  --tasks gsm8k,humaneval \
  --output_path ./results/gpt4o_baseline.json

# 새 모델 평가
lm_eval --model openai-chat-completions \
  --model_args model=gpt-4o-mini \
  --tasks gsm8k,humaneval \
  --output_path ./results/gpt4o_mini.json

# 결과 비교
python compare_results.py results/gpt4o_baseline.json results/gpt4o_mini.json
```
