# 10. 실전 활용법

> Claude Code를 더 효과적으로 사용하기 위한 실무 팁과 전략을 정리합니다.

---

## 효과적인 요청 방법

### 구체적으로 말하기

```
나쁜 예: "버그 고쳐줘"
좋은 예: "src/api/users.ts 의 getUserById 함수에서 
         ID가 없을 때 500 에러가 아니라 404를 반환해야 해"
```

Claude는 맥락이 구체적일수록 더 정확하게 작업합니다.

---

### 제약 조건 명시하기

```
"테스트 코드는 건드리지 말고 구현 파일만 수정해줘"
"기존 API 시그니처를 유지하면서 내부 로직만 바꿔줘"
"TypeScript strict 모드 위반 없이 작업해줘"
```

---

### 단계적으로 요청하기

한 번에 너무 많은 작업을 요청하면 실수가 생길 수 있습니다.

```
1단계: "auth 미들웨어 현재 구조를 분석해줘"
2단계: "JWT 검증 로직의 문제점을 찾아줘"
3단계: "이제 그 부분을 수정해줘"
```

---

### 확인 요청하기

```
"수정하기 전에 어떻게 바꿀 건지 먼저 설명해줘"
"영향받는 파일 목록부터 보여줘"
```

Claude는 실행 전에 계획을 설명하게 하면 실수를 줄일 수 있습니다.

---

## 컨텍스트 관리 전략

### 언제 /compact를 써야 하나

- `/context`로 확인했을 때 70% 이상 차있을 때
- 오래된 대화가 많이 쌓여 Claude가 초기 내용을 기억 못할 때
- 새로운 하위 작업으로 전환할 때

```bash
/compact 지금까지 결정된 API 구조와 데이터 모델만 남겨줘
```

---

### 언제 /clear를 써야 하나

- 완전히 다른 주제의 작업을 시작할 때
- 이전 대화가 오히려 방해가 될 때
- 컨텍스트가 너무 꼬여서 압축으로도 해결 안 될 때

---

### 큰 코드베이스 다루기

```bash
# 필요한 디렉토리만 추가
/add-dir ./src/api          # API 관련 작업 시
/add-dir ./src/components   # UI 관련 작업 시

# 전체 탐색은 에이전트에게 위임
> 탐색 에이전트로 인증 관련 파일만 찾아줘
```

## 비용 절약 전략

### 모델 선택

| 작업 | 권장 모델 |
|------|---------|
| 단순 질문, 코드 설명 | Haiku |
| 일반 코드 작성, 버그 수정 | Sonnet (기본값) |
| 복잡한 아키텍처 설계, 어려운 버그 | Opus |

```bash
/model claude-haiku-4-5    # 간단한 작업 전
/model claude-opus-4-7     # 복잡한 작업 전
```

---

### 불필요한 파일 읽기 줄이기

"전체 프로젝트 분석해줘" 보다는 관련 파일을 명시하세요.

```
나쁜 예: "프로젝트 전체를 분석해서 문제점 찾아줘"
좋은 예: "src/api/orders.ts 파일에서 N+1 쿼리 문제를 찾아줘"
```

---

### 캐시 효과 활용

CLAUDE.md는 세션 시작 시 항상 로드되어 캐시에 올라갑니다. 자주 참조하는 내용을 CLAUDE.md에 넣으면 토큰 절약에 도움됩니다.

```bash
/cost    # 세션 비용 확인
```

## 자주 쓰는 작업 패턴

### 새 기능 구현 워크플로우

```
1. /init (처음인 경우)
2. > [기능 설명] 구현해줘. 먼저 어떻게 할지 설명해줘.
3. 계획 검토 후 진행
4. > 이제 구현해줘
5. /review (PR 전 검토)
```

---

### 버그 수정 워크플로우

```
1. > [증상 설명]. 원인 찾아줘
2. > 영향받는 파일 목록 보여줘
3. > [원인] 이 맞아? 확인해줘
4. > 이제 고쳐줘
5. > 테스트 실행해줘
```

---

### 코드 리뷰 워크플로우

```
1. > src/api/payment.ts 파일 리뷰해줘
   또는
1. /review    (변경된 모든 파일 리뷰)
2. > [특정 지적사항] 수정해줘
```

---

### 레거시 코드 이해하기

```
1. > 이 파일이 전체 시스템에서 어떤 역할을 하는지 설명해줘
2. > 이 함수가 호출되는 경로를 추적해줘
3. > 이 코드의 잠재적인 문제점은 뭐야?
```

## 협업 팁

### 팀에서 사용할 때

**공유해야 할 것:**
- `.claude/settings.json` — 팀 공통 권한, MCP 서버 설정
- `CLAUDE.md` — 프로젝트 컨벤션, 규칙

**공유하면 안 되는 것:**
- `.claude/settings.local.json` — 개인 API 키
- `~/.claude/projects/` — 개인 메모리

**팁:** `CLAUDE.md`를 PR 리뷰 프로세스에 포함시키세요. 팀 컨벤션이 바뀌면 CLAUDE.md도 같이 업데이트합니다.

## Claude Code를 믿고 맡겨야 할 때 vs 직접 확인해야 할 때

### 맡겨도 되는 것
- 단순한 코드 수정 및 버그 수정
- 테스트 작성
- 코드 설명 및 분석
- 문서화

### 반드시 확인해야 하는 것
- 데이터베이스 마이그레이션
- 설정 파일 변경
- 외부 API 호출 코드
- 인증/보안 관련 코드
- 프로덕션 배포 명령

## 트러블슈팅

### Claude가 계속 같은 실수를 반복할 때

1. 해당 내용을 CLAUDE.md에 명시적으로 추가
2. "기억해줘"로 메모리에 저장
3. 요청을 더 구체적으로 표현

### Claude가 파일을 못 찾을 때

```bash
/add-dir ./해당경로    # 해당 디렉토리 추가
# 또는
> glob 패턴으로 찾아줘: **/파일명.ts
```

### 컨텍스트가 너무 빨리 찰 때

- 필요한 파일만 언급 (전체 분석 요청 자제)
- 작업을 작은 단위로 나누기
- `/compact`로 중간중간 압축
- 에이전트에게 탐색 위임

### Claude가 이상하게 동작할 때

```bash
/doctor    # 환경 점검
/clear     # 컨텍스트 초기화 후 재시작
/bug       # 버그 신고
```
